# 《雾下余粮》V1 游戏设计与配置架构

> 文档状态：V1 基线  
> 项目类型：轻量搜打撤资源管理 H5 Demo  
> 核心关键词：庇护所、户外狩猎、食物压力、疯狂成长、参数化 AI、数据驱动

## 1. 项目定位

《雾下余粮》是一款轻量化的“搜打撤 + 生存资源管理”游戏。

核心体验参考：

- 《逃离塔科夫》的外出搜集、风险判断和撤离。
- 《饥荒》的生存资源管理、食物压力和克苏鲁氛围。

V1 不追求复杂枪械、重度装备构筑、庞大地图或高强度生存模拟，而聚焦于一个短而完整的循环：

> 从庇护所出发，进入户外狩猎怪物，收集食物，在疯狂与生存之间做取舍，然后安全撤回庇护所。

游戏的核心不是“杀怪掉装备”，而是：

> 玩家必须食用怪物来维持生存，而食用怪物又会让玩家逐渐疯狂。

疯狂不只是惩罚。它会提高战斗能力，同时让玩家逐渐远离正常的人类状态。

## 2. V1 设计目标

V1 必须验证以下内容：

1. 庇护所到户外再回到庇护所的完整搜打撤循环。
2. 饥饿与疯狂共同驱动的资源决策。
3. 三种怪物使用同一套状态机，通过配置参数形成差异。
4. 尸体切割需要时间，切割过程会产生风险。
5. 失败会丢失本次搜集物，但不会破坏默认装备。
6. 配置页面是产品功能，而不是临时调试面板或 GM 系统。
7. 游戏逻辑依赖统一运行时配置，而不直接依赖配置页面或存储来源。

## 3. 世界结构

### 3.1 庇护所

庇护所是安全区域，承担：

- 查看玩家状态。
- 管理安全食物和怪物肉。
- 食用食物。
- 查看固定装备。
- 进行安全种植。
- 选择外出区域。
- 保存成功撤离带回的资源。

庇护所中的安全食物恢复饥饿，不增加疯狂值。V1 提供少量初始安全食物，作为玩家的安全保障。

### 3.2 户外区域

户外是主要行动区域，玩家可以：

- 探索地图。
- 遭遇并攻击怪物。
- 被怪物感知、追踪和攻击。
- 切割尸体获得怪物肉。
- 在户外食用怪物肉。
- 判断继续探索还是提前撤离。
- 到达固定撤离点并完成读条。

户外种植在架构中预留，但 V1 不实现完整生态。

## 4. 核心游戏循环

1. 玩家在庇护所检查食物、疯狂值和装备。
2. 选择进入户外区域。
3. 在地图中寻找怪物。
4. 与怪物战斗。
5. 对尸体执行切割。
6. 获得怪物肉。
7. 判断是否继续深入。
8. 必要时食用怪物肉，以疯狂换取生存和攻击能力。
9. 前往撤离点并完成撤离读条。
10. 将本次资源带回庇护所，推进种植周期并准备下一次外出。

核心决策包括：

- 当前生命和饥饿是否支持继续深入。
- 怪物收益是否值得承担战斗风险。
- 是否食用怪物肉维持行动。
- 是否接受疯狂值上升换取攻击力。
- 是否提前撤离保住资源。

## 5. 玩家系统

### 5.1 生命值

- 被怪物攻击时扣除生命。
- 饥饿归零后持续受到伤害。
- 生命归零时本次外出失败。
- V1 死亡时丢失本次获得的资源。
- 默认装备保留。

### 5.2 饥饿值

- 户外行动期间持续降低。
- 饥饿归零后持续扣除生命。
- 安全食物和怪物肉均可恢复饥饿。
- 安全食物不会增加疯狂。
- 怪物肉会增加疯狂。

### 5.3 疯狂值

疯狂值表示玩家受到异常力量污染的程度。

V1 只实现正向战斗效果：疯狂阶段越高，攻击倍率越高。

| 默认范围 | 状态 | 默认攻击倍率 |
| --- | --- | ---: |
| 0～29 | 清醒 | 1.00 |
| 30～59 | 低语 | 1.15 |
| 60～89 | 狂热 | 1.35 |
| 90～100 | 失控边缘 | 1.60 |

表现层需要暗示疯狂不是纯粹增益：

- 屏幕边缘出现异常颜色与脉动。
- 状态名称随阶段变化。
- 后续可增加低语、幻觉、错误目标和操作干扰。

V1 默认死亡后保留疯狂值，使连续外出形成长期状态压力。

### 5.4 装备

V1 提供固定默认装备：

- 一把近战武器。
- 一件基础防护装备。
- 固定切割能力或装备提供的切割倍率。

V1 不实现装备掉落、耐久、配件、词条和复杂等级。

### 5.5 背包

- V1 背包只需要承载怪物肉等核心搜集物。
- 背包容量由玩家配置提供。
- 切割收益受剩余容量限制。
- 成功撤离后，本次怪物肉转入庇护所库存。

## 6. 食物系统

### 6.1 安全食物

来源：

- 庇护所初始库存。
- 庇护所种植。
- 后续版本中的正常食物来源。

效果：

- 恢复饥饿。
- 不增加疯狂。

### 6.2 怪物肉

来源：切割怪物尸体。

效果：

- 恢复饥饿。
- 增加疯狂。
- 是 V1 户外最重要的资源。

## 7. 统一怪物 AI

怪物不拆成三套完全独立的类型代码。所有怪物共用同一套状态机，通过参数启用或禁用行为。

### 7.1 公共状态

- `Idle`：待机。
- `Wander`：随机游荡。
- `Alert`：发现可疑目标，V1 可合并到追踪切换逻辑。
- `Chase`：追踪玩家。
- `Attack`：攻击玩家。
- `Return`：放弃战斗并返回出生区域。
- `Dead`：死亡，可被切割。
- `Harvested`：已被切割。

### 7.2 核心参数

至少包括：

- 是否可以移动。
- 是否随机游荡。
- 是否主动敌对。
- 是否允许追踪。
- 感知距离。
- 丢失目标距离与等待时间。
- 最大追踪距离。
- 最大离出生点距离。
- 是否返回出生点。
- 移动、游荡、追踪和返回速度。
- 攻击距离、攻击力和攻击间隔。
- 生命值和碰撞半径。
- 是否可以切割。
- 切割时间和基础肉量。

### 7.3 三种 V1 预设

#### 静止型

- 不移动。
- 不游荡。
- 不主动攻击。
- 可被击杀和切割。
- 用于教学和低风险资源来源。

#### 游荡者

- 在出生区域附近随机移动。
- 发现玩家后主动追击。
- 超出追踪或领地限制后返回。

#### 追踪者

- 平时可以静止，也可以通过配置改为游荡。
- 感知距离和追踪距离更远。
- 超出领地或追踪限制后放弃战斗并返回。

“静止型”“游荡者”“追踪者”是配置预设，不是三套彼此隔离的 AI 实现。

## 8. 战斗系统

玩家能力：

- 移动。
- 普通攻击。
- 受击与短暂无敌。
- 死亡。
- 切割尸体。

怪物能力：

- 移动和游荡。
- 感知玩家。
- 追踪和普通攻击。
- 放弃战斗并返回出生点。
- 死亡并留下尸体。

V1 不实现技能树、连招、弹药、远程武器、部位破坏、护甲穿透和元素伤害。

## 9. 尸体、搜刮与切割

怪物死亡后留下尸体，尸体理论上包含两类资源：

1. 怪物携带物：工具、人类遗物、种子、材料或装备。
2. 怪物自身资源：肉、骨、皮、器官和污染组织。

V1 保留携带物掉落表结构，但默认掉落表为空；只实现怪物肉。

切割流程：

1. 玩家靠近尸体。
2. 长按或持续执行切割交互。
3. 显示切割进度。
4. 切割期间不能攻击或移动。
5. 完成后按配置获得怪物肉。
6. 尸体进入 `Harvested` 状态。

默认预设：

| 怪物 | 基础肉量 | 切割时间 |
| --- | ---: | ---: |
| 静止型 | 2 | 2 秒 |
| 游荡者 | 3 | 3 秒 |
| 追踪者 | 5 | 4 秒 |

切割结果预留切割速度、产量倍率、工具倍率、稀有材料概率和尸体完整度等扩展点。

## 10. 种植系统

### 10.1 庇护所种植

- 安全、稳定。
- 产出安全食物。
- 按完成外出的次数推进生长周期。
- V1 只提供一个种植格和一种作物。

### 10.2 户外种植

定位为高风险、高收益系统，后续可以加入：

- 再次进入同一区域收获。
- 被怪物破坏。
- 受到污染。
- 更高产量或更短周期。

V1 只保留配置开关与结构，不实现完整户外种植。

## 11. 撤离与失败

户外地图有固定撤离点。玩家进入范围并完成短暂读条后撤离成功。

成功撤离：

- 带回本次怪物肉。
- 保留默认装备。
- 更新庇护所库存。
- 推进种植周期。
- 记录本次结果。

撤离失败：

- 返回庇护所。
- 默认丢失本次搜集资源。
- 默认装备保留。
- 疯狂值是否保留由配置决定，V1 默认保留。

## 12. 主页面与场景入口

主页面 V1 提供两个入口：

1. **开始游戏**：读取当前生效配置，进入庇护所，再从庇护所开始外出。
2. **配置**：编辑游戏规则、内容与平衡数据。

后续可增加继续游戏、重置存档、配置方案和关于 Demo。

## 13. 配置系统定位

配置页面用于定义：

> 这款游戏按照什么规则运行。

它不是 GM 页面，不负责：

- 给当前玩家增加物品。
- 修改当前生命值。
- 直接杀死怪物。
- 生成或删除实体。
- 瞬间传送。
- 操纵当前运行中的状态机。

V1 配置保存后，在下一次开始游戏时生效，不强制热更新当前游戏。

## 14. 配置化边界

应该配置化的数据：

- 影响玩法、内容、平衡或关卡表现的数值。
- 开关、枚举、时间、距离、概率和数量。
- 玩家基础能力。
- 怪物行为、战斗和尸体参数。
- 食物、疯狂阶段和装备属性。
- 地图出生点、撤离点、刷怪点与障碍物。
- 种植周期和产出。
- 死亡与撤离规则。

不需要机械暴露的数据：

- 动画插值误差。
- 渲染层级等纯表现内部值。
- 内部重试次数。
- 与玩法和平衡无关的实现细节。

原则是：代码实现规则，配置提供规则所需的数据。

## 15. 配置层级

运行时配置统一为：

```ts
interface GameConfig {
  version: number;
  global: GlobalConfig;
  player: PlayerConfig;
  monsters: MonsterConfig[];
  foods: FoodConfig[];
  madnessStages: MadnessStageConfig[];
  equipment: EquipmentConfig[];
  maps: MapConfig[];
  farming: FarmingConfig[];
  shelter: ShelterConfig;
}
```

配置分类：

1. 全局规则。
2. 玩家属性。
3. 怪物定义。
4. 食物定义。
5. 疯狂阶段。
6. 装备定义。
7. 地图定义。
8. 种植定义。
9. 庇护所初始库存。

## 16. 配置页面

V1 采用左侧分类导航、右侧编辑区。

必须支持：

- 编辑玩家基础数值。
- 编辑三种怪物的全部行为与战斗参数。
- 编辑食物效果。
- 编辑疯狂阶段。
- 编辑装备基础属性。
- 编辑地图怪物数量、出生点和撤离点。
- 编辑切割时间和肉量。
- 编辑种植周期和产出。
- 保存并应用。
- 恢复默认。
- 导入 JSON。
- 导出 JSON。
- 基础配置校验。

第一版不要求可视化地图编辑器、配置云端存储、审批、版本回滚、多套方案对比和在线热更新。

## 17. 配置校验

至少校验：

- 必需分类是否存在。
- 数值是否为非负或大于零。
- 配置 ID 是否重复。
- 怪物、食物等引用是否存在。
- 疯狂阶段最小值是否大于最大值。
- 疯狂阶段是否重叠。
- 追踪距离是否小于攻击距离。
- 丢失目标距离是否小于感知距离。
- 玩家和怪物出生点是否超出地图边界。

## 18. 配置存储与生效

V1 使用浏览器本地存储：

- 内置默认配置保存在项目代码中。
- 用户配置保存到 `localStorage`。
- 配置键：`tiny-signal-game.config`。
- 存档键：`tiny-signal-game.save`。
- 开始游戏时优先读取已保存并通过校验的配置。
- 本地没有有效配置时使用内置默认值。

游戏运行时只依赖最终的 `GameConfig`，不关心配置来自默认数据、本地存储、导入文件或未来的云端服务。

## 21. 当前实现基线

截至 V1 首次可玩提交，已经实现：

- 主菜单、配置、庇护所、户外四个页面或运行状态。
- 生命、饥饿、疯狂和攻击倍率。
- 固定装备和简单背包。
- 三种怪物配置预设与统一状态机。
- 攻击、受击、死亡、尸体、切割和资源获得。
- 固定撤离点、撤离读条、成功与失败结算。
- 庇护所库存、食用和单个种植格。
- 分类配置表单、保存、恢复、校验、导入和导出。
- GitHub Pages 与 ChatGPT Sites 两套发布流程。

## 22. 已知简化与后续方向

当前实现仍属于验证核心矛盾的 Demo，后续可以按体验反馈决定优先级：

1. 改进玩家攻击方向、命中反馈和怪物受击表现。
2. 增加真正的疯狂负面效果，而不只提供攻击增益。
3. 加入更明确的撤离风险与地图路线选择。
4. 扩展携带物掉落和怪物自身资源。
5. 增加户外种植及区域持久化。
6. 增加配置方案、版本和分享能力。
7. 将配置默认值拆分为独立 JSON 文件或引入类型定义。

后续迭代必须继续遵守两个基础约束：

- 玩法规则和内容数据保持数据驱动。
- 配置页面只定义规则，不演变成 GM 工具。
